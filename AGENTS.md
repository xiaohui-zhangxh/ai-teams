# AGENTS.md

本文件为 **AI 编码助手与本仓库维护者的共享上下文**：说明仓库用途、目录约定、关键协作流程，并沉淀**已达成共识的决策**。多数工具会从仓库根目录读取 `AGENTS.md`，请在修改角色或工作流时同步更新相关章节。

---

## 本仓库是什么

**AI Teams** 是一套「按角色组织的提示词 + CLI 启动器」：

- 在 `roles/{语言}/` 下用 Markdown 定义各角色的人设、职责与输出规范。
- 通过 `bin/cursor-agent-as`（及同模式的 `*-as` 脚本）交互选择角色，将对应 `.md` 全文交给 Cursor Agent 等 CLI，实现**一键切换身份**。

目标不是模拟公司职级，而是让**不同会话或轮次**可以稳定扮演不同分工，提示词保持窄而可复用。

---

## 给 AI 代理的快速指引

1. **改角色**：只编辑目标 `roles/` 下的 `.md`；菜单由脚本递归扫描生成，无需改 `bin/`。
2. **新增角色**：在对应语言目录下使用两位数字前缀文件名（如 `10-新角色.md`），首行 `# 标题` 为菜单显示名。
3. **不要臆造脚本行为**：以 `README.md` 与 `bin/cursor-agent-as` 为准。
4. **重要结论落盘**：与用户就**本仓库**的范围、流程、命名或协作方式达成共识后，将条目追加到本文「决策与演进记录」（见下节维护约定）。

## 软件科技公司：固定协作流（权威说明）

以下顺序为 **分工与交接顺序**，不代表行政上下级：

**产品 → 架构 → 审查 → 实现 → 验收**

```mermaid
flowchart LR
    p(["1 产品"]) --> a(["2 架构"]) --> r(["3 审查"]) --> i(["4 实现"]) --> v(["5 验收"])
```

- **产品**：`产品策略负责人` 负责需求定义、范围裁剪、优先级与价值审视，产出可交接的 PRD 与验收标准。
- **架构**：`首席架构师` 输出技术蓝图、关键选型、非功能约束与实现指导。
- **审查**：`技术审查官` 在进入实现前质疑方案、控制复杂度并给出明确审查结论。
- **实现**：`实现负责人` 编码、补必要测试、暴露风险并交付可验证结果。
- **验收**：`技术验收专家` 对照 PRD 与验收标准做最终一致性验收。
- **并行把关**：`代码审计专家` 在实现后做深度代码审计，可与验收并行或前后衔接。

各角色文件**第二段**含带高亮的 Mermaid 图，标明该角色在上述流程中的位置。

## 自媒体运营企业：固定协作流（权威说明）

自媒体体系同样强调**固定流程与固定职责**，不保留纯行政色彩角色；按不同经营模式分别收敛为能力型角色链：

- **通用型**：`定位与内容策略 → 选题策划 → 内容制作 → 质检与合规 → 增长运营 → 商业化`
- **短视频型**：`账号策略 → 编导策划 → 影像制作 → 质检与合规 → 增长运营 → 直播商业化`
- **知识付费型**：`知识产品策略 → 引流内容 → 课程研发 → 质检与合规 → 私域成交 → 交付成功 → 数据分析`

约定：

- 角色名称优先体现**能力与交付物**，避免“创始人 / 总监 / 主理人”这类对 AI 协作帮助有限的行政称谓。
- 能力相近、交接界面不清的角色应主动合并，例如渠道/增长/社群，或拍摄/剪辑，或课程策划/教研。
- 当某企业类型存在明显的**发布前风险把关**需求时，保留 `质检与合规` 作为独立关口。
- 数据分析可独立存在，也可并入增长角色；判断标准是它是否构成一个稳定、可复用的独立交付环节。
